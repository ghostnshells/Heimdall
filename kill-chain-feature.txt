# Kill Chain Composer ‚Äî Implementation Plan

## Context

Panoptes already enriches every CVE with ATT&CK techniques, EPSS scores, and threat actor associations ‚Äî but displays them as isolated metadata on individual CVEs. The Kill Chain Composer synthesizes this data into **attack chains**: sequences of vulnerabilities spanning multiple kill chain stages that model how an adversary could move through the user's infrastructure. It answers: *"What's the most dangerous path through my stack, and what single patch breaks the most paths?"*

No new external APIs needed ‚Äî this is a pure synthesis layer on top of existing enriched vulnerability data.

## Implementation Sequence

### Step 1 ‚Äî Backend Service: `server/lib/killChainService.js` (new file)

**Core data structures:**

```js
// Maps ATT&CK technique IDs ‚Üí ordered kill chain stages
const KILL_CHAIN_STAGES = {
  initial_access:       { order: 1,  label: 'Initial Access',       techniques: ['T1190','T1133','T1566','T1078','T1189'] },
  execution:            { order: 2,  label: 'Execution',            techniques: ['T1059','T1203'] },
  persistence:          { order: 3,  label: 'Persistence',          techniques: ['T1505'] },
  privilege_escalation: { order: 4,  label: 'Privilege Escalation', techniques: ['T1068'] },
  defense_evasion:      { order: 5,  label: 'Defense Evasion',      techniques: ['T1211'] },
  credential_access:    { order: 6,  label: 'Credential Access',    techniques: ['T1212'] },
  lateral_movement:     { order: 7,  label: 'Lateral Movement',     techniques: ['T1210'] },
  collection:           { order: 8,  label: 'Collection',           techniques: ['T1005'] },
  exfiltration:         { order: 9,  label: 'Exfiltration',         techniques: ['T1567'] },
  impact:               { order: 10, label: 'Impact',               techniques: ['T1499','T1486','T1565'] },
};
// + reverse index: TECHNIQUE_TO_STAGE built at module init
```

**Algorithm ‚Äî `computeKillChains(vulnData, userAssets)`:**

1. **Classify** ‚Äî Filter vulns to `userAssets` (if set). For each CVE with `attackTechniques`, look up `TECHNIQUE_TO_STAGE` ‚Üí build `Map<stageKey, CVE[]>`. If < 3 stages have CVEs, return empty result.

2. **Build chains** ‚Äî Greedy forward walk, not exhaustive enumeration:
   - For each entry-point stage (`initial_access`, `execution`, `persistence`):
     - For each CVE in that stage:
       - Walk forward through ascending stages, at each stage picking the CVE that (a) is on a **different asset** than previous step (prefer cross-asset), or (b) has highest EPSS+CVSS if no cross-asset option.
       - Record chain if it spans 3+ stages.
   - Deduplicate by sorted CVE ID set.
   - Sort by risk score descending. Return top 20.
   - Complexity: O(E √ó S) where E = entry-point CVEs, S = 10 stages. Bounded.

3. **Score chains** ‚Äî Composite 0‚Äì100:
   ```
   (stageSpan / 10) √ó 25          // breadth
   + maxEPSS √ó 25                 // exploitability
   + (maxCVSS / 10) √ó 20          // severity
   + crossAssetBonus √ó 15         // 15 if 2+ assets, else 0
   + kevBonus √ó 10                // 10 if any step is CISA KEV
   + threatActorBonus √ó 5         // 5 if any step has named actor
   ```

4. **Break the Chain** ‚Äî Count how many chains each CVE appears in. Sort by chains disrupted. Return top 10.

**Exports:**
```js
export function computeKillChains(vulnData, userAssets = null)
// Returns: { chains[], breakRecommendations[], stageDistribution, summary }
```

**Reference:** technique IDs must align with `server/lib/attackMapping.js` (lines 5‚Äì56).

### Step 2 ‚Äî Backend Route: add to `server/index.js`

Add `GET /api/kill-chains?timeRange=X` after the `/api/vulnerabilities` route (~line 252).

- Uses `publicApiLimiter` (no auth required; auth is optional for asset filtering).
- Reads cached vuln data via `getVulnData(timeRange)` (same as `/api/vulnerabilities`).
- If auth header present, decodes JWT to get user email ‚Üí `getUserAssets(email)` for filtering.
- Caches result in DB cache with key `killchain:{timeRange}:{assetsHash}`.
- Returns `{ success: true, data: { chains, breakRecommendations, stageDistribution, summary } }`.

Pattern reference: existing `/api/vulnerabilities` route in `server/index.js`.

### Step 5 ‚Äî Frontend Components: `src/components/KillChain/` (new directory)

**5a. `KillChain.jsx`** ‚Äî Main view component

Props: `{ timeRange }`

Follow `Pulse.jsx` pattern:
- State: `data, isLoading, error, expandedChain`
- `loadData` via `useCallback` dependent on `timeRange`
- `useEffect` for initial fetch + refetch on timeRange change
- Loading state ‚Üí skeleton / spinner
- Error state ‚Üí retry button
- Empty state ‚Üí Shield icon + "No attack chains detected"
- Renders: summary stats cards, StageBar, ChainCard list, BreakTheChain table

**5b. `ChainCard.jsx`** ‚Äî Individual chain card

Props: `{ chain, isExpanded, onToggle }`

Collapsed: risk score badge (color by severity thresholds 80/60/40), stage flow as horizontal pills connected by `ChevronRight` icons, asset names below each stage, KEV/threat actor tags.

Expanded: adds full CVE detail per step (ID, asset, CVSS, EPSS, description snippet).

Stage flow uses CSS flexbox with `ChevronRight` from lucide-react between steps. No graph library.

**5c. `BreakTheChain.jsx`** ‚Äî Priority patching table

Props: `{ recommendations }`

Table columns: Priority | CVE ID | Asset | Stage | Chains Disrupted | Risk Reduction

**5d. `StageBar.jsx`** ‚Äî Stage coverage horizontal bar

Props: `{ distribution }` (`{ stage_key: count }`)

10-column CSS grid. Each column: proportional bar + stage label. Zero-count stages grayed out.

**5e. `KillChain.css`** ‚Äî All styles

Use existing CSS variables: `--severity-critical`, `--severity-high`, `--severity-medium`, `--severity-low` and their `-bg`/`-border` variants. Use `--surface-glass`, `--border-subtle`, `--radius-lg`, `--bg-tertiary`, `--text-muted`, `--space-*`, `--text-sm`. Follow Pulse.css patterns.
